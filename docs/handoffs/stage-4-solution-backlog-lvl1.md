### 2A. Reverse-ratification (do this FIRST, it constrains everything)
1. **Read the exact entries the two writers produce** (paths in boot #10). The two forms are ALREADY structurally incompatible — this is not hypothetical:
   - **opp-doc** writes **Markdown**: `## sol-A: <title>` (H2 header, **letter** id) + `### Three-slot contract` + `### Folded sub-fields` (note `target_outcome/metric` with a **slash**) + `### Reserved forward-compat fields` (still emits the **pre-collapse** `sol_state: sol-s0`). No `status`, no `readiness`, no `decision`.
   - **sol-doc** writes a **YAML code block**: `candidate_id: sol-cand-NNN-slug`, `target_outcome_metric` (**underscore**), `invariants` / `acceptance_criteria` as YAML lists, plus `status:`, `readiness:`, and a separate `decision:` axis.
   Also read the normative text in `architecture.md#Interfaces`.
2. **Pin ONE canonical serialization** for the entry — this is the stage's TOP trap (the opp-backlog build shipped *two incompatible serializations coexisting*, breaking mechanical readers). **A backlog is STRUCTURED, tabular data — ratify a STRUCTURED serialization (the sol-doc YAML-block form), NOT free-form Markdown prose subsections (opp-doc's `## sol-A` + `### subsections`).** Loose prose is the root cause of the drift: it under-constrains the write shape, so real runs diverge. Constraints: the STORE MEDIUM stays a single plain-Markdown FILE (canon invariant 1 — no JSON/DB; a YAML fenced block lives *inside* the .md file, so this stays compliant); the sol-doc YAML form is preferred because it already carries `status`/`readiness`/`decision` — the fields a store needs — which the opp-doc Markdown form lacks. Do NOT import a third form (e.g. opp-backlog's flat-bullet card layout) — that maximizes drift on the exact decision that matters most. Decide `id` scheme, field naming (`slash` vs `underscore`), field order, three-slots vs folded placement. **Design the schema forward-compatible with the lvl2 target: a base TABLE format + a Google Sheets adapter via the `gws` CLI** (stage-2 hand-back §8.1 / real-wave R4) — the user's real backlog is a Sheet, not a Markdown file; v1 stays file-based only because of prompt-only / gws-proxy-blocked, so the entry's fields must map cleanly to columns. Your SPEC OWNS the serialization; the two writers are reconciled to it (any gap = friction, §2A step 3).
3. **If the live writers DISAGREE with each other or with `architecture.md#Interfaces` → that is spec-friction**, not something you silently pick: record each drift (what opp-doc writes vs what sol-doc writes vs canon), propose the smallest reconciliation, and surface it at the STOP gate. Do NOT edit opp-doc / sol-doc / canon files (frozen for you). The build (Phase B) runs the integration check that PROVES conformance; Phase A names the target.
