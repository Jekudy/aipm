# Reference digest — grounded practice (so we don't re-derive)

> Cited facts the foundation is built on. Anti-hallucination anchor.

- Cagan/SVPG modern opportunity assessment = 4 questions asked BEFORE discovery: Objective (business objective / OKR), Key Results (measurable success), Customer Problem (from customer POV), Target Market (narrow segment) — 'enough 90% of the time' (INSPIRED 2nd ed Ch.35, oreilly.com/library/view/inspired-2nd-edition/9781119387503/c35.xhtml).
- Cagan's classic 10-question assessment is the fuller container, one Q per concern — including Q4 'what alternatives?' and Q5 'why are we best suited to pursue this?' (right-to-win, distinct from competitive landscape); an assessment 'doesn't aim to provide a solution' and must end in a go/no-go (svpg.com; maa1.medium.com toolkit-6).
- Cagan's opportunity BACKLOG holds problems/outcomes NOT committed features — that separation is what keeps it from collapsing into a roadmap; governed by 'what problem / who for / how we'll know we succeed' (svpg.com/the-opportunity-backlog/).
- Cagan's four big risks are the orthogonal quality lens every option must survive: VALUE, USABILITY, FEASIBILITY, BUSINESS VIABILITY — the categories a risks/assumptions section is scored against (productbookshelf.com/2021/01/product-discovery-techniques/).
- Amazon Working Backwards PR/FAQ press release has a fixed order (heading, subheading=segment+benefit, summary, problem-from-customer-POV, solution acknowledging 'today customers use x/y/z… those fall short', leader+customer quote, call to action); internal FAQ covers customer/TAM, problems/solutions/assumptions, economics, dependencies, and 'top-3 reasons this will NOT succeed' (workingbackwards.com/concepts/working-backwards-pr-faq-process/).
- Teresa Torres' Opportunity Solution Tree enforces the opportunity->solution SEPARATION: Outcome (a metric not a feature) -> Opportunities (customer needs, first-person, from research; an opportunity with one solution IS a solution) -> Solutions (multiple per opportunity, converge to ~3) -> Assumption Tests. This is the nested discipline inside the section container, not a competing spine (ideaplan.io/frameworks/opportunity-solution-tree; amplitude.com/blog/opportunity-solution-tree).
- Opportunity sizing formulas: Opportunity Score = Importance + max(Importance-Satisfaction,0) (Ulwick ODI); RICE = Reach*Impact*Confidence/Effort with Reach from real analytics and Confidence (100/80/50%) as a brake on weakly-evidenced bets — routinely ignored, which is the sizing-from-thin-air failure (intercom.com/blog/rice-simple-prioritization-for-product-managers/).
- GRADE ships a 4-level QUALITY ladder — High/Moderate/Low/Very-low confidence — decoupled from any readiness axis; four ordinal levels is the validated sweet spot and the anchor for the Q0-Q4 evidence-tier ladder (en.wikipedia.org/wiki/Hierarchy_of_evidence).
- TRL/CMMI are readiness/maturity ladders that measure ONLY build/adoption and DELIBERATELY exclude fitness — NASA: 'readiness does not necessarily fit with appropriateness or technology maturity.' This grounds M/Q orthogonality: M = readiness (monotonic, about existence/integration/usage), Q = fitness (can regress, about output correctness) (nasa.gov TRL definitions; cmmiinstitute.com/learning/appraisals/levels).
- Analytic (not holistic) rubric is the right tool: 3-5 levels, 4-7 criteria, one construct per row, observable descriptors (ban 'good/better/best'), a documented cut score so an unattempted section can't pass. Multi-construct rungs must be split into one-criterion-one-failure-mode items (redmenta.com analytic-vs-holistic; heqco.ca rubric guide).
- Fractal rollup: CONJUNCTIVE/weakest-link gating (level T only if ALL required criteria meet T) is the anti-slop mechanism; HEQCO explicitly warns AGAINST naive averaging across constructs because a strong section masks a hole. Use conjunctive as the gate, compensatory average only as secondary progress (heqco.ca rubric guide; emergentmind.com six-category-rating-rubric).
- EARS (Mavin, Rolls-Royce) is the machine-checkable shape: UBIQUITOUS 'The <system> shall <always>' = an INVARIANT; UNWANTED-BEHAVIOUR 'If <trigger> then the <system> shall <response>' = an error/edge acceptance criterion; Given-When-Then is isomorphic. Design-by-Contract (Meyer) distinguishes preconditions/postconditions/INVARIANTS (hold before AND after every operation) — the exact three-slot shape for the output contract (alistairmavin.com/ears/; se.inf.ethz.ch Meyer dbc_chapter.pdf).
- White-spot detection must use RETRIEVAL-as-abstention (declare ABSENT only when a scan over the actual source returns nothing) — well-calibrated — NOT the model's parametric belief that 'this is usually missing' — poorly calibrated. Distinguish ABSENT (real gap) from AMBIGUOUS (needs confirmation) globally (AbstentionBench arXiv:2506.09038).
- Skills are the correct primitive: SKILL.md name+description pre-loaded (~100 tokens), body read only when triggered — this three-level progressive disclosure is what lets the far-four ship as ~100-token M0 stubs. Package as ONE plugin; only plugin.json in .claude-plugin/, skills/ at plugin root (code.claude.com/docs/en/skills; code.claude.com/docs/en/plugins).
- State between invocations lives in FILES the PM carries, not context — 'each Claude Code session begins with a fresh context window'; skills don't share context and are ephemeral across sessions. The doc-under-construction IS the state (code.claude.com/docs/en/memory).
- TRIZ Ideal Final Result / ARIZ-85C: name the OUTCOME the system delivers 'by itself' with existing resources and no new harm, BEFORE the mechanism; backward planning (desired->current) yields higher completion (91% vs 79%) and more realistic estimates; Now-Next-Later (Bastow/ProdPad) is the native progressive-JPEG horizon — detail Now, sketch Later (triz-consulting.de ARIZ-85C; cambridge.org backward-planning JDM; prodpad.com/blog/invented-now-next-later-roadmap/).
